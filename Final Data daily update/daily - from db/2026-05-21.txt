## ⚠️ ตรงที่ต่าง — สำคัญ

### 1. V1 แยกเป็น 4-5 หมวด, V2 แยกเป็น 3 หมวด

| V1 status | จำนวน | V2 scan_type | จำนวน |
|---|---:|---|---:|
| `2` (success) | 6,590 | `success` | 6,590 |
| `4` (duplicate) | 617 | `duplicate_self` | 617 |
| `5` (not found) | 158 | `duplicate_other` | 245 |
| `6` (?) | 85 | | |
| `-100` (error?) | 2 | | |

**Mapping ที่น่าจะเป็นไปได้:**
- V1 `5+6+(-100)` = 158+85+2 = **245** ตรงกับ V2 `duplicate_other` พอดี!
- หมายความว่า V2 ตอน sync จาก V1 รวม "not_found + status6 + error" เข้าด้วยกันเป็น `duplicate_other`

### 2. Unique users ต่าง 70 คน
- V1: นับ user จาก **ทุก attempt** (รวม not_found) → 2,540
- V2 ของผม: นับ user จาก **success only** → 2,470
- ต่าง 70 = พวกที่สแกนแล้วเจอ "not found / error / status6" เท่านั้น (ไม่เคย success วันนั้น)

### 3. ที่ LINE ไม่ได้ส่งมา = "Not Found %"
LINE report เน้น **status=5 (not found)** ซึ่ง = QR ที่ไม่อยู่ใน V1 DB → ใช้ตรวจของก๊อป/ของไม่ใช่จริง
- 21 พ.ค.: not_found 158 / 7,452 = **2.1%**
- รายงานผมไม่ได้แยก not_found ออกมา → จัดเข้า "duplicate_other"

## 💡 สรุปสั้น ๆ

**ตรงกัน:** Total attempts, Success, Duplicate self — 3 ตัวเลข match เป๊ะ ๆ
**ต่างกัน:** การแบ่งหมวด "ไม่สำเร็จ" — V1 ละเอียดกว่า (not_found / dup_other / error แยกกัน) แต่ V2 รวมไว้เป็น duplicate_other หมวดเดียว

ถ้าอยากให้รายงานต่อไปแสดง **Not Found % แยก** (เหมือน LINE) ผมจะเช็คก่อนว่า V2 มี field ไหนแยกได้ หรือต้องไปดึงจาก V1 ตรง ๆ ก็ได้ — แต่ทุกตัวเลข core (success/total) **ตรงกันแน่นอน**
